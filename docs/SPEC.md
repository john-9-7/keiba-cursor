## 2. データソース

### 2.1 競馬クラスターWeb

- **URL**：[https://web.keibacluster.com/top](https://web.keibacluster.com/top)（競馬クラスターWeb β版）

#### 操作フロー（データ取得までの流れ）

1. **URLにアクセス**し、トップページを表示する。
2. **競馬種別を選択**：**中央競馬** / **南関競馬** / **地方競馬** のいずれかを選ぶ。  
   - 基本的には中央競馬を選択。場合により南関・地方を選択することもある。
3. **パスワード入力**：選択するとPW入力画面になる。その日付・種別用のパスワードを入力し、「**認証**」を実行する。
4. **会場・レースの選択**：認証後、その日の**会場ごとのページ**がある。**会場**と**レース番号**を選択する。
5. **データページ**：選択した会場・レースのページに、取得対象のデータが表示される。

**URLについて**  
- レース一覧：`https://web.keibacluster.com/top/race-list`（日付・会場を選び、1R〜12R などの一覧が表示される画面）。  
- 出馬表（取得対象）：`https://web.keibacluster.com/top/race-analyze-next`。一覧でレースをクリックするとこのURLで出馬表が表示される。  
- **レースを変えてもURLは同じ**（race-analyze-next のまま）。取得時は、表示したいレースを出馬表で表示した状態の Cookie とこのURLを使う。

#### 取得対象（スクレイピングする項目）

- **馬番**
- **RPT**（レース荒れやすさ 1～13。レース単位で1つ）
- **BB指数**（各馬の能力値）
- **リアルタイムの単勝オッズ**（同一ページに掲載。発走10分前の判定には、このオッズを発走10分前時点で取得して使う想定）

※ 上記はすべて**会場・レースを選択した先の1ページ**に載っているため、そのHTMLをスクレイピングする。

#### 取得方法（スクレイピング）

- **認証が必要**なため、以下のいずれかで実装する。
  - **ブラウザ自動化**（例：Playwright、Puppeteer）：トップ → 種別選択 → PW入力・認証 → 会場・レース選択 → 表示されたページのHTMLを解析。
  - **HTTPクライアント＋セッション（Cookie）**：認証リクエストでCookieを取得し、そのCookieを付与して会場・レースのデータURLにアクセスし、HTMLを取得して解析。
- 取得したHTMLから、馬番・RPT・BB指数・単勝オッズを**パース**して構造化データとする。
- **パース仕様（実装済み）**  
  - **race-analyze-next**: RPT は `.rpt-gauge-section .rpt-value.active`。出馬表は `#race-horses-table tbody tr` から `.horse-number-col`（馬番）、`.bb-point-col .bb-number`（BB指数）、`.win-odds-col`（単勝オッズ）を抽出。  
  - **race-analyze?race_id=**: クラスに頼らず列インデックスで抽出。td[0]=馬番、td[8]=BB指数、td[12]=単勝オッズ。データ行は td[0] が 1〜18 の行のみ。  
  - 詳細は `lib/parseRacePage.js` および [SCRAPING_SPEC.md](SCRAPING_SPEC.md) を参照。

#### Cookie（セッション）の利用

- **認証後にサーバから返るCookie**を保持し、会場・レースページへのアクセス時にそのCookieを付与すれば、認証済みとしてデータを取得できる。
- **採用する運用**：**B. ブラウザでログインしたCookieを渡す**（B案で進める）。
  - 利用者がブラウザで競馬クラスターにログインし、開発者ツールなどで取得したCookie（文字列）をアプリに貼り付ける。アプリはそのCookieを付与してデータURLにアクセスする。
  - Cookieの有効期限や取り扱い（期限切れ時の再ログイン案内等）をアプリ側で考慮する。
- （参考）A. アプリ内でログインは将来の拡張として残す。現時点では実装しない。

#### A案・B案の比較とおすすめ

| 観点 | A. アプリ内でログイン | B. Cookieを貼り付け |
|------|------------------------|----------------------|
| **メリット** | ・操作がシンプル（パスワード入力だけ）。<br>・ブラウザを開かずに完結。<br>・パスワードはアプリが送るだけなので、実装が分かりやすい。 | ・**認証の実装が不要**（ログイン画面・認証APIの解析をしなくてよい）。<br>・開発が早い。<br>・サイト側の認証仕様変更の影響を受けにくい（ブラウザでログインできればよい）。 |
| **デメリット** | ・認証リクエストの形式（URL・パラメータ・Cookie名など）を**逆解析する必要がある**。<br>・サイトの仕様変更でログインが動かなくなるリスクあり。<br>・開発・テストに手間がかかる。 | ・**毎回または定期的にブラウザでログインし、Cookieをコピーして貼る**手間がかかる。<br>・Cookieの有効期限が短いと頻繁に貼り直し。<br>・開発者ツールの操作が初心者には少し難しい場合がある。 |
| **セキュリティ** | パスワードをアプリが扱うため、保存する場合は暗号化等の考慮が必要。 | Cookieをアプリに貼り付けるため、Cookieが端末内に残る。他人に端末を触られない限りはリスクは限定的。 |
| **おすすめ** | 長期的に「アプリだけで完結させたい」なら将来Aを追加可能。 | **→ B案を採用**（本プロジェクトではBで進める）。認証の実装を後回しにでき、開発工数が少ない。 |

**※ 初心者向けの補足（「Cookieを渡す」が指す対象）**  
A・Bで言っている「Cookieを使ってアクセスする」のは**あなたが作るアプリ（競馬予想ツール）**の話です。アプリはプログラムなので、リクエストにCookieヘッダを付けて送ることができます。  
一方、**仕様書を一緒に書いているAI（こちら）**が使えるWeb取得機能は「URLを指定するだけ」で、Cookieを付けてリクエストする仕様になっていません。そのため「あなたがCookieを渡しても、AI側では認証済みページを取得できない」という説明をしています。  
→ **矛盾ではなく、「アプリならCookieでアクセスできる」「AIのツールではCookieを渡せない」という、役割の違いです。**

※ パスワードは競馬クラスター側で日ごとに発行される想定。アプリでは利用者が対象日・種別のパスワードを入力する、または安全に管理された設定で保持する方式を想定する。利用規約の確認が必要。

## 3. 共通定義

### 3.1 断層検知（全RPT共通）

- BB指数を**降順**に並べる。
- 隣り合う馬のBB指数の差で、**最初に「3.0以上」の差が開いた箇所**を「**断層**」とする。
- 断層より上＝指数が高いグループ、断層より下＝指数が低いグループとして扱う。

## 4. 判定ロジック（分析・判断ツール）

※ オッズは**レース発走10分前**のものを使用する。  
※ 実装は `lib/judgment.js` の `judgeRace(rpt, horses)` に集約。API 応答では `judgment: { verdict, pattern, lines, recommend, meta }` として返す。

### パターンA：RPT 1〜3（大荒れ・下位台頭想定）

1. **異常検知**：BB 70.0以上が1頭でもいれば全体「**見送り**」。
2. **指数足切り**：BB 44.9以下は**無条件で全除外**。
3. **グループ分け**：**主力候補**（BB 50.0〜69.9）、**爆弾候補**（BB 45.0〜49.9）。
4. **RPT細分化**  
   - **RPT1・2**：BB指数1位は過剰人気とみなしアタマ（主力）から除外し、「ヒモ（爆弾候補扱い）」へ降格。  
   - **RPT3**：上記の降格は行わず主力候補に残す。
5. **オッズ足切り**  
   - **主力候補**：一桁オッズ（単勝10.0倍未満）は除外。BB55.0以上は単勝10.0倍未満を除外。BB50.0〜54.9は単勝15.0倍未満を除外。  
   - **爆弾候補**：単勝30.0倍未満を除外（**BB1位は防波堤**としてオッズに関わらず必ずヒモ＝爆弾側に残す）。
6. **追加フィルター（合成オッズ調整）**：残った爆弾候補が4頭以上なら、BB指数が**低い順**から除外し、最大3頭まで（BB1位の防波堤は可能な限り残す）。

---

### パターンB：RPT 4〜6（アタマ上位・ヒモ荒れ想定）

1. **異常検知**：BB 70.0以上が1頭でもいれば全体「**見送り**」。
2. **断層検知**：共通の「動的断層」（§3.1）で分割。**断層より下**かつ **BB 49.9以下**は全除外。
3. **グループ分け**：**アタマ候補**（断層より上）、**ヒモ候補**（断層より下かつ BB 50.0以上）。
4. **RPT細分化**  
   - **RPT4・5**：アタマは混戦としてフラットに扱う。  
   - **RPT6**：断層上のBB1位を「**本命軸**」、他を「**連下**」とする。
5. **オッズ足切りと馬番補正**  
   - アタマ候補のうち単勝**4.9倍以下**はヒモへ降格（**RPT6の断層上BB1位**はこの降格を無視し、オッズに関わらず本命軸）。  
   - **外枠補正**：馬番が「大きい順の上位3頭（外枠3頭）」に含まれる場合は、降格ラインを**単勝5.9倍以下**に引き上げる。  
   - 残りのアタマについて、単勝**10.0倍以上**を本命、**5.0〜9.9倍**（外枠補正適用時は**6.0〜9.9倍**）を連下とする。  
   - **RPT6**では本命軸＝断層上BB1＋上記帯の本命、その他アタマは連下、ヒモは従来どおり。

---

### パターンC：RPT 7〜10（上位安定・ヒモ絞り想定）

1. **異常検知**：BB 60.0以上が**0頭**なら全体「**見送り**」。BB 49.9以下は**全除外**。
2. **グループ分け**：**絶対軸候補**（BB 60.0以上）、**ヒモ候補**（BB 50.0〜59.9）。ただしRPT7ではBB1位は概念的にヒモ扱い（軸はBB2等、下記参照）。
3. **RPT細分化**  
   - **RPT7**：BB2位をアタマ（絶対軸）。BB2位の単勝が**4.0倍未満**ならBB3位に切替（BB1位はヒモ扱い）。  
   - **RPT9**：BB1位を無条件の単独・絶対軸（1着固定）。  
   - **RPT8・10**：BB60.0以上のうち、単勝**3.0倍以上**の馬を優先して絶対軸（複数なら**オッズが高い方**）。いずれも3.0倍未満ならBB指数最大を軸とする。
4. **ぽさ検知（RPT10専用）**：BB1位が**単勝1番人気でない**場合は即「**見送り**」。
5. **オッズ足切り（ヒモ候補）**：単勝9.9倍以下は全除外。BB55.0以上は単勝10.0倍未満を除外。**15.0〜30.0倍**の中穴本線に絞る（RPT7は軸以外の馬をヒモ候補として同フィルタ）。

---

### パターンD：RPT 11〜13（ガチガチ本命決着想定）

1. **異常検知**：なし。動的断層により**断層より下は全除外**。
2. **RPT11**：超一強として断層上の**BB1位**を1着固定。BB70以上は単勝2.0倍以上、BB65以上は3.0倍以上、BB60以上は4.0倍以上を満たさなければ全体「**見送り**」（断層上BB1がBB60未満も見送り）。
3. **RPT12・13（デッドロック回避）**：断層上の馬のうち**BB1〜3位相当の3頭**に限定。3頭の単勝が**すべて3.5倍以下**なら「**見送り**」。

## 5. 出力フォーマット

- アプリは **JSON** で `verdict`（買い／見送り）、`pattern`（A〜D または ?）、`lines`（人が読む複数行テキスト）、`recommend`（推奨券種の固定文）、`meta`（馬番配列など）を返す。
- 推奨券種は **RPT に応じた固定文**をそのまま `recommend` に入れる（§5.6）。

---

### 5.1 見送りの場合（例）

```
見送り

[根拠]
（条件に応じた具体的な理由）
```

---

### 5.2 買いの場合（例・パターン別）

- **A**：主力候補・爆弾候補の馬番と推奨券種。  
- **B**：本命・連下・ヒモの馬番と推奨券種。  
- **C**：絶対軸（馬番）・ヒモ候補と推奨券種。  
- **D**：RPT11は1着固定軸、RPT12・13は断層上上位3頭相当。

---

### 5.6 推奨券種（固定文）

| RPT | テキスト（そのまま出力） |
|-----|--------------------------|
| 1〜3 | ・ワイドBOX・馬連BOX（主力候補と爆弾候補） / ・3連複フォーメーション（1列目:主力、2列目:主力＋爆弾、3列目:全候補） |
| 4・5 | ・馬単BOXまたは1着流し（アタマ候補からヒモへ） / ・3連複・3連単フォーメーション（※単勝4.9倍以下で降格した馬は2・3着付け） |
| 6 | ・馬単1着固定流し（本命軸BB1位から連下・ヒモへ） / ・3連複・3連単フォーメーション（※単勝4.9倍以下で降格した馬は2・3着付け） |
| 7 | ・馬連・ワイド少数点流し（絶対軸からヒモへ） / ・3連単逆張りフォーメーション（絶対軸を1着固定しBB1位や他ヒモへ） |
| 8・10 | ・馬連・ワイド少数点流し（絶対軸からヒモへ） / ・3連単1着固定流し（絶対軸からヒモへ） |
| 9 | ・馬連・ワイド少数点流し（BB1位からヒモへ） / ・3連単1着固定流し（BB1位からヒモへ） |
| 11 | ・単勝1点買い（BB1位） / ・3連単フォーメーション（BB1位を1着固定、残り2頭を2・3着） |
| 12・13 | ・馬連・ワイド・3連複（断層上の上位馬のみの少数点BOX） |

## 6. レース結果の取得・反映機能

- **目的**：集計分析・条件の見直しのため、実際のレース結果をアプリに取り込む。
- **取得元**：Web上からレース結果を抽出する（対象サイトは別途選定）。
- **反映内容の例**：
  - 着順、馬番、払い戻し等を取得し、保存データ（RPT・BB指数・オッズ等）と紐付ける。
  - 集計分析ツールで「RPT別・BB指数範囲別の好走状況」を計算する際に利用する。
- **実装方針**：スクレイピングまたはAPIの利用を想定。利用規約・著作権の確認が必要。

---

## 7. 集計分析ツール（概要）

- **内容**：RPT別に「どのBB指数範囲の馬が好走したか」を集計・分析する。
- **目的**：条件の見直し・ブラッシュアップ。
- **データ**：競馬クラスターから取得したデータ ＋ **Webから取得したレース結果**を紐付けて集計。
- **見せ方**：表・グラフ・ダッシュボード等、最適な形式で提案し実装する。

## 9. 今後の詰め事項

- 競馬クラスターWebの**認証後のHTML構造**：会場・レース選択先のページで、馬番・RPT・BB指数・単勝オッズがどのタグ・クラス・テーブルで囲まれているかを確認し、パース仕様（セレクタ等）を決める。※ 認証が必要なため、開発時に利用者がHTMLを保存して共有する、またはローカルでブラウザ自動化を動かして構造を取得する。
- **パスワードの扱い**：利用者入力のみとするか、設定で保存するか。保存する場合の保管方法（ローカルのみ・暗号化の要否）
- RPT・BB指数の**データ形式**（表の構造、単位、欠損時の扱い）
- **発走10分前オッズ**：同一ページのリアルタイム単勝オッズを、発走10分前時点で取得すればよいか（取得タイミングの運用をどうするか）
- **レース結果**の取得元サイト・取得項目（着順・馬番・払戻等）
- スクレイピング・**利用規約**の確認（競馬クラスターWeb）
