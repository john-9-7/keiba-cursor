# 次のステップ（開発の進め方）

仕様書（SPEC.md）ができたあと、以下の順で進めることをおすすめします。

---

## いま「何をしたらいい？」（現在地）

データ取得・パースまで実装済みです。次のどれかから選べます。

- **パターンA（ツールをそのまま使う）** … サーバー起動 → ブラウザで http://localhost:3000 → Cookie と URL を貼って「取得」。RPT・馬番・BB・オッズの表が出ればOK。判定は手元で。
- **パターンB（判定をアプリに組み込みたい）** … 「取得した RPT と horses を仕様書の判定ロジックで判定し、見送り/買い＋買い目を表示する機能を追加して」と依頼する。
- **パターンC（レース一覧から選んで詳細取得）** … 「race-list を取得して日付・会場でレース一覧を出し、レースを選んだら詳細を取得する画面を追加して」と依頼する。

---

## 今すぐの「次の行動」（B案採用後の現在地）

仕様は固まり、認証は**B案（Cookie貼り付け）**で進めることが決まっています。このあと進められるのは次のどちらか（または並行）です。

| 優先 | 行動 | 誰がやる | やると何が進むか |
|------|------|----------|-------------------|
| **1** | **技術選定＋プロジェクトの骨組みを作る** | 開発（AIと一緒に） | アプリの土台ができ、Cookie入力画面や「指定URLにCookie付きでアクセスする」処理を実装できる。 |
| **2** | **データページのHTMLを1つ保存する** | あなた | ブラウザでログイン → 会場・レースを選んでデータページを表示 → 「ページのソースを表示」や開発者ツールでHTMLをコピーし、`docs/sample_race.html` などとして保存。これを共有してもらえれば、馬番・RPT・BB指数・オッズをどう抜き出すか（パース設計）ができる。 |

**おすすめの進め方**  
- **1を先にやる**：技術選定（例：Node.js + 簡易Web画面 など）→ プロジェクト作成 → Cookie入力UIと「Cookie付きでURLにアクセスしてHTMLを取得する」まで実装。データページのURLが分かれば、取得だけは動かせる。  
- **2ができたら**：保存したHTMLを元にパース処理を追加し、「1レース分の馬番・RPT・BB指数・オッズ」を取り出せるようにする。

「次の行動」として**どちらから進めるか**決めれば、その方向で具体的な手順を出します。  
（例：「技術選定からやりたい」「まずHTMLを保存する手順を教えて」）

---

## ステップ1：前提の確認（実装前に決めること）

| やること | 説明 |
|----------|------|
| **競馬クラスターのURL・ページ構成の共有** | スクレイピングする対象のWebページ（例：〇〇の〇〇ページ）を教えてもらう。HTMLの構造が分かれば取得処理を設計できる。 |
| **レース結果の取得元の決定** | どこから着順・払戻を取るか（例：netkeiba、JRA公式など）。サイトが決まれば同じく取得仕様を決められる。 |
| **開発環境の準備** | PCに Node.js や Python など、Webアプリ開発用の環境を入れる（技術選定のあとで具体的な手順を出せる）。 |

※ 競馬クラスターのURLや「どのページにRPT・BB指数が出ているか」が分かれば、ステップ2から具体的な実装に入れます。

---

## ステップ2：技術選定

- **どんな形のアプリにするか**を決める。
  - 例：ブラウザで開くWebアプリ（PC・iPhone両対応）＋ バックエンドでデータ取得・判定。
- **使用する技術**を決める。
  - 例：フロントは React や Vue、バックエンドは Node.js や Python、データ保存は SQLite や JSON ファイルなど。
- 決まったら、プロジェクトのフォルダ構成・使用するライブラリを一覧にすると後が楽。

---

## ステップ3：データ取得の実装（最優先）

1. **競馬クラスターからデータを取る**
   - 対象ページのURL・更新タイミングを確認。
   - RPT・BB指数・馬番・オッズなどを取得する処理（スクレイピング等）を実装。
   - 取得したデータを保存する形式（JSON/DB）を決めて保存する。
2. **発走10分前オッズ**
   - 競馬クラスターに含まれるか、別サイトから取るかを決め、同じく取得処理を実装。
3. **レース結果の取得**
   - 選んだサイトから着順・馬番・払戻等を取得し、既存データと紐付けられるように保存。

※ 「まず競馬クラスターの1レース分を手動で取れるようにする」から始め、動いたら日付・開催・レース単位で拡張すると進めやすいです。

---

## ステップ4：分析・判断機能の実装

1. **判定ロジックの実装**
   - 仕様書の「判定ロジック」をそのままコードに落とす。
   - 共通の断層検知 → RPTでパターンA〜Dを切り替え → 各パターンの異常検知・足切り・グループ分け・オッズ足切り。
2. **出力の実装**
   - 仕様書の「出力フォーマット」に合わせて、「見送り」または「買い」＋買い目・推奨券種を表示する。
3. **画面・操作**
   - 対象の日・開催・レースを選ぶ → 取得データで判定 → 結果を表示する画面を作る。

---

## ステップ5：集計分析機能の実装

1. **データの紐付け**
   - 保存している「予想時点のデータ（RPT・BB指数等）」と「レース結果」を、日付・開催・レースで紐付ける。
2. **集計ロジック**
   - RPT別に、BB指数の範囲（例：45〜50、50〜55…）ごとに「何頭が〇着以内だったか」などを集計する。
3. **見せ方**
   - 表・グラフ・簡易ダッシュボードなど、仕様書の「集計分析ツール」に沿った画面を用意する。

---

## ステップ6：PC・スマホで使いやすくする

- 画面をレスポンシブにして、PC・iPhoneの両方で見やすくする。
- 必要なら「ホーム画面に追加」でPWAとして使えるようにする。

---

## すぐできること（今の段階で）

1. **競馬クラスターの情報を整理する**
   - 利用しているページのURL、どのあたりにRPT・BB指数が載っているか（表なのか、テキストなのか）をメモする。
2. **レース結果はどこから取るか決める**
   - 例：netkeiba、JRAの結果ページなど、候補を1つに絞る。
3. **開発環境の希望を決める**
   - 「できるだけシンプルに」「Pythonでやりたい」「JavaScriptでWebアプリにしたい」など、希望があれば伝える。

    ↓
