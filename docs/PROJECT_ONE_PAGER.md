# 競馬予想ツール プロジェクト1枚要約

## 1. 目的（何を実現したいか）

- 競馬クラスターの **RPT・BB指数・オッズ** を自動取得する。
- あなた独自ルールで **買い / 見送り** と **買い目** を自動判定する。
- 予想時点データと実結果を蓄積し、**有効条件を継続改善**できる仕組みにする。
- 実運用で止まらないよう、**高速・堅牢・再現可能**な運用にする。

## 2. 運用フロー（当日〜改善）

1. 当日データ取得  
   Cookie認証で一覧・詳細を取得。会場ごとに対象レースを自動選定。

2. 自動判定  
   RPT帯ごとのロジック（BB・オッズ・断層・例外条件）を適用し、判定を出力。

3. 蓄積  
   取得時点の情報を保存。`race_id` 重複はスキップしてデータ品質を守る。

4. 結果突合  
   実着順を取り込み、BB順位との対応を記録する。

5. 改善ループ  
   RPT別・BB帯別の傾向を見て条件を更新し、次開催で再検証する。

## 3. 現時点の方針

- **速度優先**: 発走済みレースは基本スキップし、当日判断に集中する。
- **安定優先**: 変更時は回帰テストとスモークテストで破壊を早期検知する。
- **運用優先**: 本番反映前後のチェックをワンコマンドで回せる状態を維持する。

## 4. 成功条件（KPIイメージ）

- 運用速度: レース前判断までの時間を短縮できること。
- 安定性: 変更後も主要導線（取得・判定・表示）が壊れないこと。
- 検証可能性: 後から条件の有効性を説明できること。
- 再現性: 本番反映手順と確認手順が毎回同じで回せること。

## 5. 直近の優先順位

1. 当日取得と判定の安定運用（止まらないこと）
2. 結果突合データの品質向上（分析できる形に揃える）
3. 分析画面の強化（改善判断を早くする）
4. ルール改修のしやすさ向上（変更コストを下げる）

## 6. この文書の使い方

- 新しい要望を入れる前に、この1枚に照らして優先順位を決める。
- 実装の判断に迷ったときは「当日運用の速度・安定性」を優先する。
- 条件ロジックを更新したら、検証手順（回帰・スモーク）までセットで実施する。
