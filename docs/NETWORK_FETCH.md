# 競馬クラスターへの取得（安定性）

## 方針

- **出馬表ページ（`race-analyze?race_id=`）への HTTP 取得**は **`lib/keibaAnalyzeFetch.js` に集約**しています。
- **データ蓄積（`lib/raceAccumulator.js`）**とはコードを分離し、蓄積機能の変更がダッシュボードの取得経路に直接触れないようにしています。
- クラウド（Render 等）からの同時アクセスで相手サーバーが **500 / 429** を返すことがあるため、**5xx・429 時は自動リトライ**（短い待ち時間付き）します。

## 環境変数（任意）

| 変数名 | 既定 | 説明 |
|--------|------|------|
| `DASHBOARD_FETCH_CONCURRENCY` | `1` | ダッシュボードでの**同時取得数**（1=直列・500 対策、2〜4 で並列化可能）。 |
| `DASHBOARD_FETCH_DELAY_MS` | `800` | レース取得**間の待機**（ms）。サーバー負荷軽減・500 回避用。 |
| `DASHBOARD_INITIAL_DELAY_MS` | `1200` | 一覧取得後、最初のレース取得**前の待機**（ms）。連打を避ける。 |
| `ACCUMULATE_FETCH_CONCURRENCY` | `3` | 一括蓄積時の同時取得数。 |

## 500 回避の仕組み

- **指数バックオフ＋ジッター**：リトライ時の待機を 1→2→4 回目で指数増加し、ランダム 0〜400ms を加算
- **リクエストタイムアウト**：15〜18 秒で打ち切り、リトライに回す
- **ブラウザ寄りヘッダ**：`Sec-Fetch-Dest` / `Sec-Fetch-Mode` を付与
- **Referer**：レース一覧から遷移した想定で `race-list` を指定

## 発走済みレースのスキップ

**発走時刻を過ぎたレース**は詳細取得をスキップします（カードには「発走済みのためスキップ」と表示）。処理の高速化と、発走後のサーバー応答不良（500 等）の回避を目的としています。

## 画面上の「HTTP 500」について

カードに **HTTP 500** と出る場合、多くは **競馬クラスター側が 500 を返している**状態です。

1. しばらく待って **更新** または **再取得** を試す（リトライで改善することがあります）。
2. まだ出る場合は **`DASHBOARD_FETCH_DELAY_MS=1200`** や **`DASHBOARD_INITIAL_DELAY_MS=2000`** に増やす。
3. Cookie の取り直し、競馬クラスター側のメンテナンス有無を確認する。
